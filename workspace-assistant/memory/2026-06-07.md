### 2. 上下文架构对齐 DeepSeek（核心改动，commit 7b9f5b0）
**关键纠正**：06-03 以为"随机 user 能让 Gateway 无状态"——今天实测发现 Gateway 有三层：
- **会话层**（按 user/sessionId 隔离）✅ 隔离正常，随机 user 能让它自带历史=空
- **workspace 记忆层**（MEMORY.md，webchat-v4/opus 共享）← 真正的"串台"来源：模型用 write 工具自己把测试数字 5566/8812/4823 写进了 workspace-webchat/MEMORY.md，每次新会话注入
- **session-memory hook**（全局，只在 /new 时归档）❌ 无关

**最终方案（管理员定）**：
- 发消息用**随机 user**（`_randomUser()`），WS + HTTP 两路都改 → Gateway 会话层每次空白，消除"塞两遍"
- `loadRecentHistory` 改**全量带历史 + 150K token 兜底**（HISTORY_TOKEN_BUDGET=150000，_estTokens=字符/2），替换原"最近20条"。理由：DeepSeek 1M 窗口个人对话根本到不了上限，全量最忠实；兜底按最小模型 Opus 200K 保守取 150K
- 切模型不丢上下文 = 免费附赠（历史跟 session 文件走，不跟模型走）
- 清空 MEMORY.md 测试污染（只清数字，保留写记忆能力）

- 目标：①对话框→卡片喵(自然语言) ②展开态勾选框→REST，**两种都改 data 都刷新显示**

- 卡片喵是被动 agent，不需实时感知前端改动，每次被调用先读 data 即可（同"上下文以文件为准"哲学）
