## 🐛 修复：网页登录后 1 秒内被踢回登录页

**现象**：管理员反馈网页登录成功后立即退回登录界面。
**根因**：前端全局 fetch 拦截器（index.html ~4021行）对任意 401 即 `clearAuth()`+`showLoginOnly()`。Dashboard 卡片请求（`loadCardData` 等）从 `#token` 输入框取 Gateway Token，但网页账号登录的用户该框为空 → 发出空 `Bearer ` → 拦截器误判"已带 Authorization"不注入 JWT → 服务端返回 401 → 被踢出。日志印证：`空 Token...401` 与带 JWT 的 `200` 交替刷屏。

**修复（/var/www/chat/index.html，commit 41bd17e 已 push origin main）**：
1. 拦截器治本：空 Bearer 视作无凭证 → 用 JWT 覆盖注入。
2. 源头修正：卡片请求 token 改 `authJwt || #token.value`（29 处）。
3. `connect()`（连 Gateway）保持读 #token，未动。

已同步 repo + 更新 ac-design-code-audit-V2.0（AU-FIX-1）。备份：backups/index.html.bak-loginfix-20260611-105644。

**教训**：全局 401→登出 的拦截器必须区分"凭证真过期"vs"某个子请求碰巧没带对凭证"，否则一个无关请求的 401 会误踢整个会话。

## 🔥 事故：todo 卡片请求风暴(27次/秒) → 网页加载慢20s+/failed to fetch

**现象**：管理员报网页加载极慢(20s+)、偶发 failed to fetch、卡在"正在连接服务器"。
**排查**：服务端全链路(Gateway/nginx/Tunnel/chat completions)实测均 200 正常 → 问题在客户端。统计 fileserver 请求发现 `display/todo` 2808次/2分钟(27次/秒)，大量 401。
**根因(死循环)**：
1. `todo/data.json` 是旧嵌套格式 `{user:{daily:[]}}`，但 `generate-display.py` 期望扁平 `{user:[...]}` → 脚本崩溃(`'str' object has no attribute 'get'`)
2. 脚本崩 → display.json 的 `generated_date` 永远停在旧日期(06-09)
3. 后端 `_maybe_refresh_timesensitive`(todo rules time_sensitive=true) 每次读取判定"过期" → `_regenerate_display`(脚本再崩) 但仍执行 `_broadcast(card_changed)`
4. WS 推 card_changed{todo} → 前端 handleWSEvent → loadTodoCard → 再读 display/todo → 回到第2步 → **无限风暴**

**修复**：
1. todo/data.json 转扁平格式 + generate-display.py 加容错(兼容旧嵌套+过滤非dict)，脚本不再崩，generated_date 正确更新 → 断掉循环源
2. 前端 loadTodoCard/loadGenericCard 加 800ms 节流锁(纵深防御，防任何卡片风暴) commit 2b4cf5e

**教训**：
- **time_sensitive 卡片 + generate-display 崩溃 = 刷新风暴炸弹**：_regenerate_display 即使脚本失败也广播 card_changed，前端收到就重载→再触发。应在 _broadcast 前确认 display 真的更新成功(generated_date==今天)，否则别广播。(待加固 fileserver)
- **data_schema 一致铁律**：旧嵌套数据没迁移到新扁平格式 = 定时炸弹。改 schema 必须迁移存量数据。
- 卡片加载统一加节流，防御性必需。
- 排查"加载慢"先统计服务端请求频次(journalctl + uniq -c)，一眼看出失控循环。
