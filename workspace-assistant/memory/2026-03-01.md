## 学习前辈经验
用户发来 `clawdwise-knowledge-pack.zip`，这是一个运行了数周的飞书群聊 AI Bot 的完整知识沉淀。

### 核心经验教训（LESSONS.md 精华）
1. **不确定就说不知道** — 编造答案代价大
2. **犯错先查自己** — 不甩锅给系统
3. **断言前先查证** — 涉及事实先召回确认
4. **替换前先确认新的能工作** — 不删旧方案在新方案未验证时
5. **说到做到立即执行** — 不留"待做"
6. **用户反复追问先承认情绪**
7. **中文 Embedding 选 m3e-base** (MRR 0.844) 远超 all-MiniLM-L6-v2 (MRR 0.375)
8. **多路召回 > 单路** — grep+KG+FTS+Vector 四路并行
9. **subagent 处理长任务** — 主 session 不做超过3轮 tool call 的任务
10. **新建 pipeline/cron 必须更新文档**
11. **git commit 包含事实** — Before/After、命令输出，不写模糊描述
12. **配置文件用追加不用覆写**
13. **社工检测必须有** — 群聊用户会尝试注入攻击
14. **trash-put 替代 rm**
15. **敏感操作先通知 owner**

## 学习第二位前辈经验：多Agent + GitHub协作实战指南
来源：喵仔（PM喵）的 multi-agent-guide.md

### 核心架构：Issue驱动的异步协作
- **Issue = Agent间的异步API**：创建Issue = 发请求，Comment = 回调，Close = 确认完成
- 对比实时聊天：有记录、可追踪、天然持久化，15分钟延迟完全可接受
- 5只AI喵管理6个私有仓库，累计关闭200+ Issue

### 角色分离铁律
- **PM不写代码，开发不创建Issue** — 所有改动必须有Issue追踪和验收
- PM（喵仔）：规划Epic、拆Issue、分配标签、巡检进度
- 开发喵：读Issue → 写代码 → commit → push → comment
- 领域喵（Moka/知识/日报）：项目规划、创建Issue、验收代码、Close Issue
- 采集喵：执行搜索+抓取+入库

### SOUL vs MEMORY 分离
- **SOUL = 不变的身份和规则**（几乎不改，改了就是重大决策）
- **MEMORY = 会变的状态和经验**（频繁更新）
- 分离原因：避免频繁更新状态时误改铁律

### 心跳驱动的自治闭环
- 错峰调度：开发喵:00, 日报喵:03, 知识喵:05（避免资源冲突）
- 闭环：规划→创建Issue→开发→提交→验收→Close→规划下一个
- **心跳不是签到，是找活**：open Issue < 2 → 必须创建新Issue
- 每个项目始终保持2-3个open Issue（Issue池空了 = 项目停滞）

### Issue铁律
1. 必须有验收标准（无标准 = 垃圾Issue）
2. PM不直接close Issue（只有领域喵验收通过后才能close）
3. 一个Issue只做一件事
4. Issue是持久化的工作记录
5. 一个Issue只给一个子Agent（避免超时丢失进度）
6. 子Agent超时统一30分钟

### 模型选择原则
- **复杂工作用Claude Opus**：代码、分析、规划、验收
- **简单运维用DeepSeek**：日报收集、邮件扫描、跑脚本
- **效果 > 成本**：先确保好用，再考虑省钱
- 不确定用什么模型时，默认Claude Opus

### Top 10经验教训
1. PM不写代码 — 哪怕改一个变量名也要走Issue流程
2. 不确定的事不瞎编 — 编造看似合理的信息危害大100倍
3. 每个Issue必须有验收标准
4. 旧代码必须及时清理 — 新功能上线 = 旧代码同步删除
5. 搜索摘要不是知识 — 必须Jina Reader抓全文，每条≥200字
6. 一个Issue只给一个子Agent
7. 并行开发要 pull --rebase — 绝不force push
8. 公开仓库不用AI Agent操作 — 所有操作必须人类审核
9. 心跳不是签到是找活 — 主动找活而非被动等活
10. 效果第一成本其次
